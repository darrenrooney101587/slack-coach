Slack Coach Interactions

Three distinct interaction flows handled by socket_server.py over a persistent WebSocket connection: recap review approval, message feedback voting, and next-topic poll voting. All state is persisted to local JSON files in STATE_DIR.

01

Overview

How it works: The Slack coach interactive layer listens for button clicks from Slack users over a persistent Socket Mode WebSocket connection (via the coach-socket container). When a user clicks any button in a coach-posted message, Slack dispatches an action event to the running SocketModeHandler, which routes it to the appropriate handler by action_id.

There are three independent interaction flows. Review mode lets a designated reviewer approve or skip a held recap before it goes public — the recap is stored in held_recaps.json until actioned. Feedback voting (thumbs up/down) records per-user sentiment on a posted message and updates the button label in-place. Topic poll voting lets users vote for the next coaching topic and sees live vote counts and avatars refresh on the message.

All state is written to JSON files under STATE_DIR — not a database. A volume mount shared between coach-server and coach-socket is required for review mode to work (the server writes held recaps; the socket server reads and pops them).

02

Review Mode

When REVIEW_MODE=true, an incoming meeting recap is held rather than posted immediately. hold_recap() writes the Block Kit blocks and target channel_id to held_recaps.json under a fresh UUID key. send_review_dm() then posts the full recap preview to the reviewer's DM, appending Approve and Skip buttons whose value is the UUID.

When the reviewer clicks a button, the Socket Mode handler receives the action. pop_recap(recap_id) removes the entry from held_recaps.json atomically — a second click on the same button finds nothing and logs a warning. On approve, post_recap() sends the blocks to the original target channel. In both cases, _update_dm_status() replaces the DM message with a one-line status.

Shared volume required: coach-server writes to held_recaps.json; coach-socket reads and pops from it. Both containers must mount the same STATE_DIR. If the socket container restarts between hold and approve, the file persists — but if the server container has a different mount path, the pop will silently find nothing.
03

Feedback Voting

Thumbs up and thumbs down handlers follow the same two-phase pattern. The vote is recorded immediately on the first call to record_vote() — even before the user's avatar is fetched. A second call with user_image populated updates the same entry if the users.info API call succeeds. This means every vote counts even if the avatar fetch fails.

Votes are written to feedback_{job}_{channel}.json. The key is the Slack message timestamp (message_ts). If a user votes again, their existing vote entry is overwritten — users can change their vote, but cannot vote twice. After recording, the handler calls chat.update to mutate the button text in-place: the chosen button shows a checkmark; the unchosen shows the opposite emoji.

Vote overwrite semantics: each user has exactly one vote per message. Clicking thumbs down after thumbs up replaces the earlier vote. The record_vote loop finds an existing entry by user_id and overwrites it — there is no append-only audit log.
04

Topic Poll Voting

The next-topic poll renders one button per candidate in the Slack message using action IDs of the form vote_next_topic_N. The handler uses a regex pattern vote_next_topic_\d+ to match any index. After recording the vote to votes_{job}_{channel}.json, the handler re-inspects the full blocks array from the incoming event body to discover all candidates and their associated context blocks.

It then calls get_poll_details() which reads the votes file and returns per-candidate vote counts and the three most recent voter avatars. Each candidate's context block is rebuilt and a single chat.update replaces the entire message with refreshed counts. This makes every vote a live read-update cycle against the state file.

Every vote is a file read: get_poll_details opens and parses the entire votes JSON file on every button click. With many voters on a high-traffic message this is fine at current scale, but becomes a contention point if multiple users click simultaneously — last writer wins on the JSON file write.
05

Dry-run behavior

No dry-run mode is implemented for any of the three interaction flows.

There is no --dry-run flag, environment variable, or code path that suppresses state writes or Slack API calls during testing. Every button click in a real Slack workspace immediately writes to the JSON state files and calls chat.update or chat.postMessage.

Testing safely: use a dedicated test Slack workspace and a separate STATE_DIR to avoid polluting production state. Review mode approval sends a real chat.postMessage to the target channel — there is no way to rehearse it without it posting.
06

State Storage

All interaction state is persisted as JSON files in STATE_DIR (default /state). Filenames encode scope: the job and channel fields from the action meta allow separate files per coaching job and channel, so vote tallies from different jobs do not collide.

File Written by Read by Contents
held_recaps.json server.py → hold_recap() socket_server.py → pop_recap() Map of UUID → {blocks, channel_id, held_at}. Entries are removed atomically on approve or skip. Shared between coach-server and coach-socket — requires common volume mount.
feedback_{job}_{channel}.json socket_server.py → record_vote() socket_server.py → get_vote_counts() Map of message_ts → {votes: [{user_id, vote, user_image, timestamp}]}. Thumbs up/down votes only. One entry per message; one vote per user (overwritten on re-vote).
votes_{job}_{channel}.json socket_server.py → record_vote() socket_server.py → get_poll_details() Map of message_ts → {votes: [{user_id, candidate, vote, user_image, timestamp}]}. Next-topic poll votes only. One entry per message; one vote per user (overwritten on re-vote); candidate field distinguishes choices.
Single-instance assumption: JSON file writes are not atomic across processes. Running multiple coach-socket replicas without an external lock would cause lost updates on concurrent votes. The current architecture assumes one socket container instance.
