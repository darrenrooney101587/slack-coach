Event-driven pipeline

Fireflies → Slack

When a meeting ends, Fireflies posts a webhook. The Flask server fetches the full transcript via GraphQL, formats it as Slack Block Kit blocks, resolves the routing target from routing.yml, and posts the recap as a DM or channel message.

event-driven flask webhook receiver fireflies graphql routing.yml config slack block kit multi-recipient
Quick read
1
Single HTTP request per meeting. No queue, no retry loop — if the webhook fails, Fireflies will not redeliver.

01

Overview

How it works: A meeting ends in Zoom or Google Meet. Fireflies records and transcribes it, then fires a meeting.summarized webhook to the configured endpoint. The Flask server at /webhooks/fireflies receives the event, extracts the meeting_id, and calls the Fireflies GraphQL API to fetch the full transcript including the AI-generated summary and action items.

The transcript is formatted into Slack Block Kit blocks — a header with the meeting title, a summary section, and per-person action items grouped by name. The server then consults routing.yml to determine the target channel or DM recipients based on the meeting title or organizer email. If no rule matches, the recap posts to the configured default_channel. When multiple targets are resolved, the recap posts to each serially.

No delivery guarantee. Fireflies does not retry failed webhooks. If the server is down or returns a non-200 response when the webhook fires, that recap is lost. The FIREFLIES_WEBHOOK_SECRET env var is optional — without it, the signature check is skipped and any caller can trigger a fetch.

03

Transcript fetch and formatting

The server calls the Fireflies GraphQL API with the meeting_id. The response includes the meeting title, summary overview, bullet points, and action items. If the transcript has no usable summary fields, the server returns 422 and stops. Otherwise format_recap() builds the Block Kit payload: a header block, divider, summary section, per-person action items, and a trailing divider.

Routing

resolve_channel(transcript, config) walks the rules list in order and returns the first match as a list of target IDs. Rules match on title or organizer_email using Python re.search() (case-insensitive). A rule can specify a single channel or a channels list for multi-recipient delivery. If no rule matches, the default_channel is returned as a one-item list.

05

Delivery

The server iterates the resolved targets list and calls post_recap(blocks, channel_id, token) for each. If REVIEW_MODE=true, the recap is held in the state directory and a review DM is sent to REVIEWER_USER_ID instead. The reviewer approves or skips via Slack buttons handled by the socket server.

06

Dry-run behavior

No dry-run mode is implemented. Every valid webhook triggers a live Fireflies GraphQL API call and a live Slack chat.postMessage call. There is no flag or environment variable to suppress delivery while exercising the pipeline.

For local testing without waiting for a real Fireflies event, use webhook_test.sh with a real meeting_id from your Fireflies account and the server running locally. This will perform a real transcript fetch and post a real Slack message.

Test against a real meeting. webhook_test.sh hits a live endpoint and posts live Slack messages. Route test traffic to a dedicated test channel (#test-dm) via routing.yml to avoid noise in production channels during development.

Reference

Components and external dependencies touched by the pipeline.

Component Role Notes
app/server.py Webhook receiver Flask app. Handles ingress, orchestrates fetch → format → route → post.Runs in coach-server container, port 8080.
app/fireflies.py Fireflies API client GraphQL POST to api.fireflies.ai/graphql. Fetches title, summary, action items.Requires FIREFLIES_API_KEY.
app/formatter.py Block Kit formatter Converts transcript dict to Slack blocks. Parses **Name** headers for per-person action items. Converts **bold** to *bold*.
app/router.py Routing resolver Walks routing.yml rules, returns list of target IDs. Supports channel (single) and channels (list) per rule.Config is cached after first load — restart required for changes.
routing.yml Routing config YAML file at project root. Baked into Docker image at build time. Rules match on title or organizer_email via regex.
app/slack.py Slack poster chat.postMessage via HTTPS. Accepts channel IDs (C...) or user IDs (U... — Slack opens a DM).Requires SLACK_BOT_TOKEN.
app/review.py Hold state (optional) Writes recap JSON to STATE_DIR when REVIEW_MODE=true. pop_recap reads and deletes it on approval or skip.
